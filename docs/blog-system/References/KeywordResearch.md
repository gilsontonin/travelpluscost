# Keyword research + the keyword map (travel-adapted)

_Reverse-engineered from Jono's keyword module, adapted for a BROAD travel site. The upstream of the whole
blog system: it produces the **keyword map** (the priority queue of what to write next) that the scan →
research → write pipeline consumes._

## The strategy (owner-set — read this first)
**Go BROAD. KD is a difficulty LABEL, never a gate.** We deliberately write across every difficulty band,
because the two halves work together (hub-and-spoke):
- **Big juicy pillars (KD ~50–70+)** = the **anchor posts**. They may take *years* to crack page 2–3 — that's
  fine. Their job is **topical authority**: covering the important, high-volume topics tells Google we're a
  real authority on the city, not a thin one-trick site.
- **Easy keywords (KD <30)** = the **spokes**. They rank early *and* eventually, AND they **link up to the
  pillars**, passing authority — which is what eventually helps the slow pillars rank. **A rising tide lifts
  all ships.**

So we are NOT limiting to KD ≤30. We take the juicy shots too. (Jono's KD≤30-only rule is for *local service
SEO*, where precision matters; travel is broad — we want coverage + authority.)

## The two page types
- **Blog (informational/commercial intent)** — the guides. The keyword map below is mostly these.
- **Money (transactional)** — for us these are the **hotel property pages + `/hotels/<city>` hubs** (auto-
  generated, region-generalized) and the **"where to stay in <city>" program** (`References/InventoryPosts.md`
  — our version of Jono's service×city "zipper"). We don't hand-build a service-page matrix; we have it.

## The levers (ballpark, not precision)
Source = the **downloaded Semrush broad-match CSVs** in `Designs/Locations/<slug>_broad-match_*.csv`. That
data is good enough to ballpark — **no Google Ads Keyword Planner** (his $1-for-exact-volumes trick is local-
service precision we don't need going broad).
- **Intent** — keep Informational + Commercial for blog; drop pure Navigational/Transactional.
- **Volume** — ≥100/mo to bother; **lead prioritization by volume (opportunity), take the juice.**
- **KD** — a band label, NOT a filter: **🟢 <30 · 🟡 30–49 · 🟠 50–69 · 🔴 70+.** Write across all.
- **CPC** — ignored (local-service signal).

## Exclusions (baked into `scripts/blog/lib/keywords.mjs`)
Never target: **near-me** (a Google signal, not a keyword — ranking for it = spam flag), jobs/"how to
become", competitor + OTA brands (Expedia/Booking/etc.), branded-navigational ("homewood suites french
quarter", "X tickets"), adult, merch/posters/tv-guide, and unrelated verticals (sports/real-estate/civic/
weather-utility). The shared `NOISE` + `BRANDED` regexes enforce this for both robots.

## Clusters + hub-and-spoke
- **Cluster:** one **primary** keyword + a cluster of **similar secondary** keywords per post (5 shots, not 1).
- **Hub-and-spoke:** the pillar (hub, the juicy anchor) ↔ the spokes (the easy <30 posts). Spokes link UP to
  the pillar and the pillar links down to them — passing authority both ways. Wire these cross-links when you
  publish (we already do it across the NOLA cluster).

## The tools (the workflow)
1. `npm run blog:cluster -- "<city>"` — explore: buckets the CSV into travel topics (head, volume, KD,
   winnable count, seeds). `--topic <name>` dumps one cluster.
2. `npm run blog:map -- "<city>"` — **the keyword map**: clusters similar keywords around a primary per topic,
   bands by pillar KD (🟢/🟡/🟠/🔴), shows the **pillar (the shot) + quick-win entries (the spokes) + the
   cluster**, cross-references `content/keywords.json` (published → DONE), and writes
   **`content/keyword-map-<slug>.md`** (the priority queue, biggest opportunity first within each band).
3. **Pick the next from the map** → scan (`blog:scan`/`blog:serp`) → research (`ResearchBrief.md`) → write.
4. `npm run blog:keywords` — **cannibalization** guard (don't rank two pages for the same term; if Googling
   both shows the same SERP, merge). Log each shipped cluster to `content/keywords.json`.

## Output files
- **`content/keywords.json`** — the ledger of *published* clusters (primary + secondaries + status).
- **`content/keyword-map-<slug>.md`** — the *forward* priority queue (what to write next), regenerated by
  `blog:map`. This is Jono's "keyword map." Pull the next post from here.
